### 2.7 Pagination

- **Mechanism:** Offset-based (`limit` + `offset`)
- **Default:** `limit=20`, `offset=0`
- **Session ordering:** `ORDER BY started_at DESC` (default) or `ORDER BY last_active DESC` (when `order_by_last_active=True`)
- **Message ordering:** `ORDER BY id` (AUTOINCREMENT = true insertion order)
- **Compression chain projection:** Sessions with `end_reason='compression'` are projected forward to their latest continuation tip.

### 2.8 Message Roles

Observed roles in the codebase:
- `user` — User messages
- `assistant` — Model responses
- `tool` — Tool execution results (with `tool_call_id`, `tool_name`)
- System messages handled internally but stored in the messages table

### 2.9 Message Content Format

- Plain text for simple messages.
- Structured content (multimodal) encoded with `\x00json:` prefix via `_encode_content()` / `_decode_content()`.
- Multimodal content example:
  ```python
  [{"type": "text", "text": "Hello"}, {"type": "image_url", "image_url": {"url": "..."}}]
  ```
- Tool calls stored as JSON in `tool_calls` column.

### 2.10 Sensitive Fields in Session Data

| Field | Classification | Action |
|-------|---------------|--------|
| `system_prompt` | **Prohibited** | Never return to WebUI |
| `model_config` | **Redact** | May contain internal markers; return only safe subset |
| `billing_provider` | **Redact** | Don't expose billing details |
| `billing_base_url` | **Prohibited** | Internal API endpoint |
| `billing_mode` | **Redact** | Don't expose billing configuration |
| `user_id` | **Redact** | May contain platform identifiers; return only if safe |
| `cwd` | **Redact** | May leak local paths; return truncated or omit |
| `content` (messages) | **Caution** | May contain user PII; return as-is for session view |
| `tool_calls` | **Caution** | May contain sensitive tool parameters; sanitize |

### 2.11 Session Read Side Effects

- `SessionDB.__init__` with `read_only=True` opens the database in read-only mode (`mode=ro` URI parameter).
- **No writes occur during read operations.**
- SQLite WAL mode allows concurrent reads without blocking writers.

### 4.4 Context Loader

**Function:** `load_memory_context(query, *, max_categories=3, max_memories=5, max_record_chars=3000, include_archived=False) -> MemoryContextResult`

**Scoring:**
- Categories scored by keyword/term match against query.
- Memories scored within selected categories.
- No LLM involvement in scoring.
- Purely text matching + keyword matching.

**Safety:**
- **No side effects.** The function reads files and computes scores in-memory.
- **No persistence.** Results are not written anywhere.
- **No LLM calls.** Scoring is algorithmic.

## 10. Known Technical Debt

1. **SessionDB lacks cursor-based pagination.** All pagination is offset-based. This is acceptable for a dev-only read-only API but may need improvement later.

2. **No DTO layer exists.** Current web_server.py returns raw dict objects from SessionDB directly. The Dev WebUI API must introduce a proper DTO layer for safety.

3. **`list_sessions_rich()` returns all session columns.** The API must filter to a whitelist of safe fields.

4. **CLI commands have embedded logic.** `dev-check` and `dev-info` have some business logic in the CLI layer. The API should call underlying services directly.

5. **Memory scoring is not configurable externally.** Scoring weights are hardcoded in `memory_router.py`.

## 11. Risks

| Risk | Severity | Mitigation |
|------|----------|-----------|
| Accidental exposure of system_prompt via session DTO | **P0** | DTO whitelist; never include system_prompt field |
| SessionDB read-only mode not enforced | **P1** | Always use `read_only=True`; integration test to verify |
| Memory record contains PII | **P1** | API returns summary only by default; full record requires explicit request |
| Dev API port conflicts with dev gateway (18080) | **P2** | Use a different port (e.g., 5181) |
| FastAPI CORS misconfiguration | **P1** | Explicit allowlist of `http://127.0.0.1:5180` only |

---

## 12. Conclusions

1. **All data needed by the Dev WebUI is accessible through existing Python service functions.** No new data access layer is required.

2. **FastAPI is already a project dependency.** The Dev WebUI API can use FastAPI without introducing new frameworks.

3. **A new independent Dev API server is the recommended approach**, separate from both the production gateway and the existing web dashboard server.

4. **All read operations are side-effect-free.** Session reads, memory reads, and context preview are pure queries.

5. **The critical safety boundary is the DTO layer.** Raw database objects must never reach the frontend.

6. **No LLM calls are needed for any Phase 0C read-only API.** The context preview uses algorithmic scoring, not model inference.
